# 07 — Reproducibility

This document defines how any experiment in this repository can be exactly reproduced by a third party: which dependencies are used, how seeds are managed, what the expected directory structure is, and how runs are launched.

Tracking artifacts and logging frequencies are defined in `adr/0005-logging-framework.md` and `adr/0006-experiment-tracking.md`. The configuration schema is defined in `12_config_schema.md`. This document assumes those decisions as given and focuses on the mechanics of reproduction.

---

## Execution Targets

Every experiment must be reproducible under three equivalent modes:

- Local installation via [uv](https://docs.astral.sh/uv/), with `uv.lock` guaranteeing exact versions.
- Docker container on CPU (default profile).
- Docker container on NVIDIA GPU (`gpu` profile), used for PPO and DQN at full budget.

The three modes produce identical results for the same seed because the sources of randomness (environment, agent, network initialization) are controlled explicitly.

---

## Dependency Management

### Toolchain

- [uv](https://docs.astral.sh/uv/) as package manager and resolver.
- `pyproject.toml` declares direct dependencies following [PEP 621](https://peps.python.org/pep-0621/).
- `uv.lock` pins the full transitive graph with `SHA-256` hashes and is versioned in the repository.

### Rationale

`uv` was chosen over `pip + requirements.txt` and over `poetry` for three reasons:

- Deterministic resolution with hash verification (`--require-hashes` equivalent) by default.
- Speed: $10 \times$ to $100 \times$ faster than `pip` in cold installs, relevant for CI and Docker builds.
- Compatibility with the standard `pyproject.toml`, portable to any tool that speaks PEP 517/621.

### Pinned Versions

| Component | Version | Reason |
| --- | --- | --- |
| Python | $3.12$ | Active support, compatible with PyTorch 2.3+ and Stable-Baselines3 2.3+ |
| PyTorch | $2.3.1$ | Latest stable release with official wheels for CUDA 12.1 and CPU |
| Gymnasium | $0.29.x$ | Current Farama API, supported by SB3 2.3 |
| Stable-Baselines3 | $2.3.x$ | Compatible with PyTorch 2.3 and Gymnasium 0.29, provides DQN and PPO |
| NumPy | $<2.0$ | SB3 2.3 does not yet support NumPy 2 |
| Pandas | $2.2.x$ | Consumes `logs.csv` for statistical analysis |
| Matplotlib | $\geq 3.8$ | Generates the plots defined in `09_results_template.md` |

The `uv.lock` file is the authoritative source. This table is informative.

### Local Installation

```bash
uv sync
uv run python -m scripts.verify_install
```

`uv sync` installs exactly the versions pinned in `uv.lock`. `verify_install` prints the installed versions of `torch`, `gymnasium`, `stable_baselines3`, and checks CUDA availability if applicable.

## Containerization

Two Docker images are defined, with a shared base for everything not related to compute runtime.

### Naming Convention

All containers and images generated by this project use the prefix `evrl-` (short for evolutionary-vs-rl-lab) to avoid collisions with other projects running simultaneously on the same host:

| Artifact | Name |
| --- | --- |
| CPU image | `evrl-trainer-cpu` |
| GPU image | `evrl-trainer-gpu` |
| CPU container | `evrl-trainer-cpu` |
| GPU container | `evrl-trainer-gpu` |
| Network | `evrl-net` |
| Volumes | `evrl-runs`, `evrl-cache` |

This is the same convention used across Syntel Labs infrastructure and makes it trivial to identify what each container belongs to via `docker ps`.

### Image Structure

- `docker/Dockerfile.cpu`: base `python:3.12-slim`, installs `uv`, copies `pyproject.toml` and `uv.lock`, runs `uv sync --frozen`.
- `docker/Dockerfile.gpu`: base `nvidia/cuda:12.1.1-cudnn8-runtime-ubuntu22.04`, installs Python 3.12 via `deadsnakes`, installs `uv`, and performs the same `uv sync --frozen`.

Both images mount the source code as a read-only volume, so the container is not rebuilt for every code change.

### Docker Compose

The services are orchestrated in `docker/docker-compose.yml` following the [n8n-stack](https://github.com/Syntel-Labs/n8n-stack) pattern: YAML anchors for shared configuration and explicit profiles (`cpu` / `gpu`) for runtime selection.

```yaml
x-trainer-base: &trainer-base
  image: evrl-trainer:dev
  volumes:
    - ../src:/app/src:ro
    - ../configs:/app/configs:ro
    - ../runs:/app/runs
  environment:
    - PYTHONHASHSEED=0

services:
  trainer-cpu:
    <<: *trainer-base
    container_name: evrl-trainer-cpu
    build:
      context: ..
      dockerfile: docker/Dockerfile.cpu
    profiles: ["cpu"]

  trainer-gpu:
    <<: *trainer-base
    container_name: evrl-trainer-gpu
    build:
      context: ..
      dockerfile: docker/Dockerfile.gpu
    profiles: ["gpu"]
    deploy:
      resources:
        reservations:
          devices:
            - driver: nvidia
              count: all
              capabilities: [gpu]
```

### GPU Requirements

- NVIDIA driver $\geq 535$.
- [NVIDIA Container Toolkit](https://docs.nvidia.com/datacenter/cloud-native/container-toolkit/latest/install-guide.html) installed on the host.
- CUDA 12.1 compatible GPU (compute capability $\geq 7.0$).

Tested in the reference environment on RTX 30-series and 40-series. GPU usage is optional: the project runs fully on CPU, only slower.

## Seeds and Determinism

### Fixed Seed Set

The main experiment uses $N = 10$ seeds, identical for all methods:

$$
\mathcal{S} = \{11,\ 23,\ 37,\ 53,\ 71,\ 89,\ 101,\ 127,\ 149,\ 173\}
$$

These values are small primes spaced across the first two hundred natural numbers. The choice is arbitrary but documented here as immutable: any future experiment must reuse this set so that results remain directly comparable.

### Seed Derivation

Each run derives three independent seeds from the base seed $s \in \mathcal{S}$:

| Seed | Formula | Controls |
| --- | --- | --- |
| `env_seed` | $s$ | Environment stochasticity (obstacles, initial state) |
| `agent_seed` | $1000 + s$ | Network initialization, exploration, sampling |
| `eval_seed` | $2000 + s$ | Intermediate evaluations without exploration |

The separation prevents correlation between environment and agent from biasing results.

### Required Seed Calls

At the start of each run, before any stochastic operation:

```python
import random, numpy as np, torch
random.seed(agent_seed)
np.random.seed(agent_seed)
torch.manual_seed(agent_seed)
torch.cuda.manual_seed_all(agent_seed)
env.reset(seed=env_seed)
```

Additionally, in Docker and CI: `PYTHONHASHSEED=0` as environment variable.

### CUDA Determinism Disclaimer

By default PyTorch on GPU is not strictly deterministic due to non-deterministic kernels in convolutional and reduction operations. For comparisons across seeds this does not affect the study because the variance introduced is smaller than the variance between seeds. If strict determinism is required:

```python
torch.use_deterministic_algorithms(True)
torch.backends.cudnn.deterministic = True
torch.backends.cudnn.benchmark = False
```

This mode has a performance cost of roughly $10$ to $30\%$ and is not enabled by default.

## Execution via Makefile

All reproduction operations go through `make` targets, following the pattern established in `n8n-stack`. Scripts in `scripts/` may be invoked by `make` when the logic does not fit cleanly in a single line.

### Expected Targets

| Target | Description |
| --- | --- |
| `make install` | `uv sync` locally |
| `make build-cpu` | Builds the CPU image |
| `make build-gpu` | Builds the GPU image |
| `make train METHOD=ppo SEED=11` | Runs a single run locally |
| `make train-all METHOD=ppo` | Runs the ten seeds of a method |
| `make experiment-main` | Complete main experiment ($40$ runs) |
| `make experiment-robustness` | Robustness tests as defined in `05_experimental_design.md` |
| `make aggregate` | Runs `aggregate.py` and produces the comparative table |
| `make tensorboard` | Opens TensorBoard on `runs/` |
| `make clean` | Removes temporary artifacts without touching `runs/` |
| `make clean-runs` | Removes `runs/` completely, requires manual confirmation |

### Invocation Examples

```bash
make install
make train METHOD=dqn SEED=23
make experiment-main
make aggregate
```

For Docker:

```bash
make build-cpu
docker compose --profile cpu run --rm trainer-cpu make train METHOD=ga SEED=11
```

## Configuration Snapshots

At the start of each run, the runner writes `runs/{experiment}/{method}/seed_{s}/config.json` with the full configuration actually used, including defaults that were not overridden. This file is the source of truth for reproducing the run: any difference between two executions with the same `config.json` and the same seed is a bug.

The structure of `config.json` is defined in `12_config_schema.md`.

## Full Reproduction Procedure

To reproduce the main experiment from a clean clone:

1. Install prerequisites: `git`, `make`, `docker` with compose plugin. NVIDIA toolkit if GPU is used.
2. Clone the repository and enter its directory.
3. `make install` to install local dependencies (optional if everything will run in Docker).
4. `make build-cpu` or `make build-gpu` as appropriate.
5. `make experiment-main` to launch the $40$ runs. The runner respects the overwrite protection from ADR 0006: already completed runs are skipped.
6. `make aggregate` to produce the final comparative table from `runs/registry.json` and each `metrics.json`.
7. The generated table is directly consumable by the plotting scripts defined in `09_results_template.md`.

Total expected wall-clock time on a mid-range GPU: approximately $8$ to $12$ hours for the main experiment. On CPU: $24$ to $48$ hours depending on the machine.

## Reproducibility Checklist

Before declaring an experiment reproducible, all the following must hold:

- [ ] `uv.lock` committed and up to date with `pyproject.toml`.
- [ ] Docker images build without warnings on a machine other than the developer's.
- [ ] Two independent runs with the same seed produce identical `metrics.json` (CPU) or within numerical noise (GPU without strict mode).
- [ ] `registry.json` correctly reports the state of all runs.
- [ ] Every plot in `09_results_template.md` can be regenerated from `runs/` without manual intervention.
- [ ] The `verify_install` script passes in all three modes (local, CPU Docker, GPU Docker).
