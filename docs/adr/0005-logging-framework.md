# ADR 0005 — Logging Framework

## Status

Accepted

---

## Context

It was necessary to decide how to log metrics, artifacts, and the state of each experiment during training and evaluation, ensuring full traceability, reproducibility, and compatibility with the statistical analysis defined in `06_metrics_definition.md`.

The central problem was choosing between an existing logging solution or building a custom one, and defining what is logged, how frequently, and in what format.

Active constraints at the time of the decision:

* Compatibility with GA, DQN, and PPO under the same interface
* Logging of comparable curves on the same environment steps axis
* Support for multiple seeds without file collisions
* Format readable by pandas for subsequent statistical analysis
* No overhead that significantly affects training time

---

## Options Considered

### Option 1 — Weights & Biases (wandb)

Advantages: real-time visualization, automatic comparison between runs, cloud storage.

Discarded because it introduces dependency on an external service, requires an account and internet connection, and may not be available in all execution environments. Reproducibility should not depend on external services.

### Option 2 — MLflow

Advantages: full experiment tracking, local UI, artifact storage.

Discarded because it adds unnecessary infrastructure complexity for the scope of the project. It requires a local or remote server and additional configuration that does not contribute to the comparative objective.

### Option 3 — TensorBoard only

Advantages: standard visualization in PyTorch, integration with SB3.

Discarded as a standalone solution because TensorBoard event files are not directly readable by pandas for statistical analysis. It is kept as an additional visualization format but not as the primary data source.

### Option 4 — Fully custom logging from scratch

Advantages: full control over every logged field.

Discarded because it reinvents basic functionality available in Python’s standard library, introduces risk of bugs in the logging system, and consumes development time that does not contribute to the experimental objective.

### Option 5 — Structured CSV + JSON summary + TensorBoard

Advantages: CSV directly readable by pandas, JSON for per-run summary, TensorBoard for visualization during training, no external dependencies, simple and auditable format.

**Selected.**

---

## Decision

A three-layer complementary logging system is used:

**Layer 1 — Structured CSV** (`logs.csv`)

Primary record for subsequent statistical analysis. Written every $10{,}000$ steps for all methods.

**Layer 2 — JSON summary** (`metrics.json`)

Final summary per run at the end of the budget. Includes all metrics defined in `06_metrics_definition.md`.

**Layer 3 — TensorBoard** (events)

Visualization during training. Automatically generated by SB3 for DQN and PPO. Manually implemented for GA.

### File structure per run

```bash
runs/
  {method}/
    seed_{s}/
      config.json     — exact configuration snapshot
      logs.csv        — step-level metrics
      metrics.json    — final summary
      events.out.*    — TensorBoard
      checkpoints/
        step_250000.pt
        step_500000.pt
        ...
```

### CSV fields

| Field          | Description                       | Methods  |
| -------------- | --------------------------------- | -------- |
| `step`         | Current environment step          | All      |
| `episode`      | Episode number                    | All      |
| `reward`       | Episode reward                    | All      |
| `reward_ma`    | Moving average $W=100$            | All      |
| `wall_time`    | Elapsed seconds                   | All      |
| `loss`         | Network loss                      | DQN, PPO |
| `epsilon`      | Current exploration value         | DQN      |
| `q_mean`       | Mean Q-value                      | DQN      |
| `entropy`      | Policy entropy                    | PPO      |
| `generation`   | Current generation                | GA       |
| `best_fitness` | Best fitness of the generation    | GA       |
| `mean_fitness` | Mean fitness of the generation    | GA       |
| `eval_reward`  | Reward in intermediate evaluation | All      |
| `eval_step`    | Step at which evaluation occurred | All      |

### JSON summary fields

```json
{
  "method": "dqn",
  "seed": 11,
  "env_seed": 11,
  "agent_seed": 1011,
  "total_steps": 2000000,
  "wall_time_seconds": 0,
  "converged": true,
  "steps_to_threshold": 0,
  "r_final": 0.0,
  "r_final_std": 0.0,
  "best_reward": 0.0,
  "auc": 0.0,
  "config_snapshot": {}
}
```

### Logging frequency

| Event                   | Frequency               |
| ----------------------- | ----------------------- |
| Step-level metrics      | Every $10{,}000$ steps  |
| Intermediate evaluation | Every $50{,}000$ steps  |
| Model checkpoint        | Every $250{,}000$ steps |
| JSON summary            | At end of run           |
| Config snapshot         | At start of run         |

### Conversion of GA curve to environment steps

To maintain the same horizontal axis as DQN and PPO, the GA curve is logged in accumulated steps:

$$\text{steps}*g = g \times N*{\text{pop}} \times \bar{T}_{\text{ep}}$$

This value is written in the `step` field of the CSV so that all curves are directly comparable on the same axis.

---

## Consequences

### Positive

* No external dependencies: reproducible in any environment
* CSV directly consumable by pandas for statistical analysis
* Predictable and auditable directory structure
* TensorBoard available for monitoring during training
* Config snapshot guarantees that each run is fully reproducible from its own artifacts
* Same format for all methods facilitates comparative analysis

### Negative

* No real-time cloud-based visualization
* CSV can grow significantly with $2\text{M}$ steps and $10$ seeds per method
* GA TensorBoard requires manual implementation, unlike DQN and PPO which get it automatically via SB3

---

### Impact on other components

The `step` field in the CSV as a common axis is the integration point with `06_metrics_definition.md`. All metrics defined there are computed from this file.

Checkpoints saved every $250{,}000$ steps allow reconstruction of the full learning curve and restarting experiments from intermediate points without repeating the full budget.

The config snapshot in `config.json` at the start of each run guarantees that the exact hyperparameters are linked to each result, satisfying the requirements of `07_reproducibility.md`.
